WiFi Server

The TeenAstro WiFi Server runs on ESP8266 and provides a TCP bridge exposing the LX200 serial protocol over WiFi. Clients (Flutter app, ASCOM, Stellarium) connect to the server and send/receive LX200 commands as if they were on a serial port.

Source: TeenAstroServer/, libraries/TeenAstroWifi/

TCP Bridge (Port 9999)

The server listens on TCP port 9999. Each connected client receives a transparent bridge to the MainUnit serial line:

  • Client sends LX200 commands (:CMD#) over TCP
  • Server forwards commands to MainUnit via serial (57600 baud)
  • Server receives replies and forwards them back to the client
The WiFi Server does not interpret commands; it acts as a transparent serial-to-TCP bridge.

Web Configuration UI

The server hosts a web interface for WiFi configuration:

  • Station mode: SSID and password for connecting to an existing WiFi network
  • Access Point mode: SSID and password for the ESP8266’s own hotspot
  • Web password: protects the configuration pages
  • Site and mount settings: forwarded to MainUnit via LX200

Configuration is served over HTTP; form submissions update EEPROM and WiFi settings.

Bounded Input Handling (strncpy)

All web form input is copied with bounded functions to prevent buffer overflows:

InputHandling
webpwdstrncpy(masterPassword, v.c_str(), sizeof(masterPassword)-1) + null termination
wifi_sta_ssidstrncpy(wifi_sta_ssid[k], v.c_str(), sizeof(wifi_sta_ssid[k])-1)
wifi_sta_pwdstrncpy(wifi_sta_pwd[k], v.c_str(), sizeof(wifi_sta_pwd[k])-1)
wifi_ap_ssidstrncpy(wifi_ap_ssid, v.c_str(), sizeof(wifi_ap_ssid)-1)
wifi_ap_pwdstrncpy(wifi_ap_pwd, v.c_str(), sizeof(wifi_ap_pwd)-1)
No unchecked strcpy of server.arg() into fixed buffers was found in the audit.

TeenAstroWifi Library

The TeenAstroWifi library provides:

  • TCP server on port 9999
  • Serial bridge: TCP ↔ MainUnit serial
  • Web server for configuration pages
  • mDNS for discovery (optional)

Location: libraries/TeenAstroWifi/

readBuffer / writeBuffer Sizes and Safety

TCP command handling uses fixed-size buffers with explicit bounds checks:

BufferSizeSafety
writeBuffer50 byteswriteBufferPos > 49 checked; buffer reset on overflow
readBuffer130 bytessendReceiveAuto(..., readBuffer, sizeof(readBuffer), ...) — bounded

The GXAS reply is 136 base64 chars + #. Clients connecting to the WiFi server receive the full reply; the bridge forwards bytes transparently. Internal buffers are bounded to prevent overflow.
